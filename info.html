Introduction to Information Disclosure

Information disclosure vulnerabilities occur when a website unintentionally reveals sensitive information to its users. This information can range from technical details about the web application to business-critical data that could be exploited by attackers.

Common Information Disclosure Vulnerabilities

Server Version Disclosure

Web servers often reveal their version numbers and other technical details through HTTP headers, error messages, or default pages.

Impact

Attackers can use version information to:

  • Search for known vulnerabilities in specific versions
  • Launch targeted attacks with version-specific exploits
  • Identify potential attack vectors (e.g., PHP version might suggest certain vulnerabilities)

Example

HTTP/1.1 200 OK
Server: Apache/2.4.29 (Ubuntu)
X-Powered-By: PHP/7.2.24
X-AspNet-Version: 4.0.30319

This reveals the exact Apache version, operating system, PHP version, and ASP.NET version.

Tools to Discover

Nmap
Nuclei
Nikto
httpx-toolkit
Shortscan

Exploitation for RCE/DDoS

With version information, attackers can:

  • Search for CVEsCommon Vulnerabilities and Exposures affecting that specific version
  • Exploit known vulnerabilities to gain remote code execution
  • Identify services vulnerable to amplification attacks for DDoS
Example Nmap Command 
nmap -sV --script=banner,http-server-header,http-title -p80,443 target.com

This command will identify service versions and grab HTTP headers that might reveal server information.

Sensitive Data Exposure

Applications sometimes expose sensitive directories, files, or information that should be protected.

Directory Listings

When directory listing is enabled, attackers can browse through files and find sensitive information.

http://example.com/images/  # Lists all files in the directory

Backup Files

Common backup file extensions that might contain source code or credentials:

http://example.com/.bak
http://example.com/index.php~
http://example.com/db.sql.bak

Configuration Files

Files like .env, config.php, or web.config might contain database credentials or API keys.

http://example.com/.env
DB_USER=admin
DB_PASS=SuperSecret123!

Version Control

Exposed .git, .svn, or .hg directories can reveal source code and commit history.

http://example.com/.git/
http://example.com/.svn/entries

AWS S3 Bucket Exposure

Misconfigured AWS S3 buckets can expose sensitive business data, user information, or application assets.

Impact

  • Data breaches exposing millions of user records
  • Exposure of proprietary business information
  • Financial loss due to data theft or ransomware
  • Reputation damage and regulatory fines

Discovery Methods

Bucket Naming Conventions

Common patterns like:

  • {company}-prod
  • {company}-backup
  • s3-{company}-logs
  • assets-{company}

JavaScript Analysis

Frontend JavaScript often contains S3 bucket URLs:

https://s3.amazonaws.com/{bucket}/
https://{bucket}.s3.amazonaws.com/

Error Messages

Applications might reveal bucket names in error messages when resources are missing.

Tools to Discover

lazys3
awscli
s3scanner
java2s3
Nuclei

Example s3scanner Command

s3scanner --bucket company-name-backup --region us-east-1

This will check if 'company-name-backup' exists and its permissions.

Checking Bucket Permissions 
aws s3api get-bucket-acl --bucket bucket-name
aws s3api get-bucket-policy --bucket bucket-name

These commands will show who has access to the bucket and what permissions they have.

Original IP Disclosure

Even when protected by CDNs like Cloudflare, the original server IP might be exposed through various methods.

Impact

Bypassing CDN protection allows attackers to:

  • Directly target the origin server
  • Bypass WAF and rate limiting
  • Exploit vulnerabilities that the CDN might mitigate

Discovery Methods

Historical DNS Records

Before the site used a CDN, it might have pointed directly to the origin IP.

SecurityTrails
ViewDNS
Wayback Machine

SSL Certificates

Scanning IP ranges for SSL certificates matching the target domain.

Censys
Shodan

Subdomains

Some subdomains (dev, staging, test) might point directly to the origin.

subfinder
assetfinder

Email Servers

Email headers or SPF records might reveal internal IPs.

Example Shodan Search

ssl:"target.com" port:443

This will find servers with SSL certificates for target.com, which might include the origin server.

Advanced Discovery Techniques

Automated Scanning Workflow

A comprehensive approach to information disclosure testing:

1. Initial Reconnaissance 
subfinder -d target.com -o subs.txt
httpx -l subs.txt -title -status-code -tech-detect -o httpx_results.txt

Discover subdomains and gather basic HTTP information.

2. Sensitive File Discovery 
nuclei -l httpx_results.txt -t exposures/ -severity medium,high,critical

Scan for common information disclosure patterns.

3. Bucket Discovery 
cat subs.txt | lazys3 -output buckets.txt
s3scanner --list buckets.txt --dump

Find and analyze potential S3 buckets.

4. Origin IP Discovery 
shodan search --fields ip_str,port ssl:"target.com"
censys search "services.tls.certificates.leaf_data.subject.common_name: target.com"

Search for origin servers bypassing CDNs.

Mitigation Strategies

Server Headers

  • Remove or obfuscate server version information
  • Configure web server to minimize header information
  • Use security modules like mod_security for Apache

Sensitive Files

  • Disable directory listings
  • Implement proper access controls
  • Remove backup files from production
  • Add .git, .env to .htaccess deny rules

AWS S3 Buckets

  • Set buckets to private by default
  • Implement bucket policies carefully
  • Enable S3 bucket logging
  • Use IAM roles instead of access keys

Origin Protection

  • Restrict origin server to CDN IPs only
  • Use different hostnames for admin interfaces
  • Implement firewall rules to block direct access
